# Simplify: 代码审查与清理

审查所有变更文件的可复用性、质量和效率。修复发现的任何问题。

## 阶段 1: 识别变更

运行 `git diff`（如果有暂存更改则运行 `git diff HEAD`）查看变更内容。如果没有 git 变更，审查用户提及的或你在此对话中早些时候编辑的最近修改的文件。

## 阶段 2: 并行启动三个审查智能体

使用 ${AGENT_TOOL_NAME} 工具在单条消息中并发启动三个智能体。向每个智能体传递完整的 diff，使其拥有完整的上下文。

### 智能体 1: 代码复用审查

对于每项变更：

1. **搜索现有的工具函数和辅助函数**，它们可能替代新编写的代码。在代码库的其他地方查找相似的模式——常见位置包括工具目录、共享模块以及变更文件相邻的文件。
2. **标记任何重复现有功能的新函数。**建议使用现有的替代函数。
3. **标记任何可以使用现有工具函数的內联逻辑**——手写的字符串操作、手动路径处理、自定义环境检查、临时类型守卫以及类似模式都是常见的候选对象。

### 智能体 2: 代码质量审查

审查相同的变更中是否存在 hack 式模式：

1. **冗余状态**：重复现有状态的状态、可以派生的缓存值、可以直接调用的观察器/副作用
2. **参数泛滥**：向函数添加新参数，而不是泛化或重组现有参数
3. **复制粘贴并稍作修改**：应该通过共享抽象统一的近似重复代码块
4. **抽象泄漏**：暴露应该封装的内部细节，或破坏现有的抽象边界
5. **字符串类型代码**：在代码库中已存在常量、枚举（字符串联合）或品牌类型的地方使用原始字符串
6. **不必要的 JSX 嵌套**：不添加布局值的包装器 Boxes/元素——检查内部组件 props（flexShrink、alignItems 等）是否已经提供了所需的行为
7. **不必要的注释**：解释代码做什么的注释（命名良好的标识符已经做到了这一点）、叙述更改的注释，或引用任务/调用者的注释——删除；只保留非显而易见的 WHY（隐藏约束、微妙不变量、变通方法）

### 智能体 3: 效率审查

审查相同变更的效率：

1. **不必要的工作**：冗余计算、重复文件读取、重复网络/API 调用、N+1 模式
2. **错过的并发性**：独立操作顺序运行，而本可以并行运行
3. **热路径膨胀**：添加到启动或每个请求/每次渲染热路径的新阻塞工作
4. **重复的无操作更新**：在轮询循环、间隔或无条件触发的事件处理程序中的状态/存储更新——添加变更检测守卫，以便在没有任何变更时不通知下游消费者。另外：如果包装函数接受更新器/归约器回调，验证它是否遵守相同引用返回（或无论"无变更"信号是什么）——否则调用者的提前返回无操作会被悄然破坏
5. **不必要的存在性检查**：在操作前预先检查文件/资源存在性（TOCTOU 反模式）——直接操作并处理错误
6. **内存**：无界数据结构、缺少清理、事件监听器泄漏
7. **过度宽泛的操作**：在只需要部分内容时读取整个文件，在过滤单个项目时加载所有项目

## 阶段 3: 修复问题

等待所有三个智能体完成。汇总它们的发现并直接修复每个问题。如果发现是误报或不值得处理，记录下来并继续——不要与发现争论，直接跳过。

完成后，简要总结修复了什么（或确认代码已经干净）。
